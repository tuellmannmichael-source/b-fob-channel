Phase unwrap — naive vs robust (predictive + WLS refit)

Mirrors cs_phase_unwrap.c logic in JavaScript for exploration. Firmware uses the same two-pass predictive scheme.

What changed and why

The legacy unwrap walks tone-by-tone and folds each difference into (−π, π]. That is naive to frequency gaps (it assumes the phase jump is only from range, not from missing bins) and propagates one bad tone into every later sample—so one multipath hit can tilt the whole phase-vs-frequency line used for slope ranging.

  • Robust method: estimate a global dφ/df from many pairwise wraps (median), then predict each sample’s branch using that slope and the actual Δf, adding integer multiples of 2π only where needed. A WLS refit weighted by tone quality updates the slope once, then the predictive step repeats.
  • Use case win: in BLE CS, tones are not always contiguous or equally trustworthy; this keeps the unwrapped phase locally consistent with physics so the subsequent weighted regression / IRLS in cs_phase_slope_estimate() starts from a cleaner line—fewer bogus distance pulls for keyfob / cabin multipath.
  • Safety: if geometry is degenerate, firmware falls back to the old sequential unwrap so you never get a hard failure from the new block alone. This page is synthetic; always validate on-air against your calibration and KPIs.
Not a security boundary. Better unwrap reduces range bias; it does not stop relay or pairing attacks. Keep cryptographic and policy controls separate.
